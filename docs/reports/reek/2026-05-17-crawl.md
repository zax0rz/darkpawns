### MED-001: `pkg/ai/` is entirely dead code — whole package never imported → DP-168
**File:** pkg/ai/brain.go:27, pkg/ai/behaviors.go, pkg/ai/ (entire package)
**What:** The full `ai` package (brain.go + behaviors.go) is never imported anywhere in the codebase. `Brain.Update()` is a literal `// Stub implementation` — returns nil, does nothing. `Behavior` interface only has `Name()` — AggressiveBehavior doesn't attack, WanderingBehavior doesn't wander, SentinelBehavior doesn't guard. Mob AI doesn't run.
**Why it matters:** The package declares itself as "implements mob AI behaviors" but no mob ever uses it. Mobs are supposed to have AI-driven behavior (aggression, wandering, sentinel). That system exists only on paper. Every mob that's NOT aggressive/wandering/sentinel — that's all of them — is effectively brain-dead. They stand still and never move or react unless commanded.
**Source:** analysis — dead code / stub
**Suggested fix:** Wire the ai package into the game loop (zone dispatcher or mob tick), implement actual behavior logic (moveToRoom for wandering, attackPlayer for aggressive), or remove the package as misleading dead code.

### LOW-001: 6 unused clamp functions in affect_manager.go → DP-171
**File:** pkg/engine/affect_manager.go:579-631
**What:** `statClampPC()`, `statClampNPC()`, `statClamp()`, `hitrollClamp()`, `damrollClamp()`, `armorClassClamp()` — 6 exported clamp functions with zero callers. C source does stat clamping inline with `MAX(3, MIN(18, value))`.
**Why it matters:** These were written as helpers for the affect pipeline but never hooked in. If someone modifies affect application logic, they'll expect clamping to happen — it won't.
**Source:** staticcheck (U1000), golangci-lint (unused)
**Suggested fix:** Wire into the stat-modification path (affectApplyImmediate / removeAffectImmediate) or remove.

### LOW-002: pkg/validation — `SanitizePlayerName`, `ValidateInput`, `ValidateCommand` are dead exports → DP-170
**File:** pkg/validation/validation.go:43, pkg/validation/input.go:43,99
**What:** `SanitizePlayerName()`, `ValidateInput()`, `ValidateCommand()` are exported but never called from outside the validation package. `SanitizePlayerName` is only called in tests. `ValidateInput` and `ValidateCommand` are only self-referenced within input.go.
**Why it matters:** Player name sanitization doesn't actually happen — `IsValidPlayerName` is used (reject invalid), but no one ever calls `SanitizePlayerName` to clean input. If an invalid name somehow reaches the system, there's no sanitization layer.
**Source:** analysis — dead export
**Suggested fix:** Wire `SanitizePlayerName` into the player creation flow, or remove the dead exports and deprecation comment.

### LOW-003: pkg/optimization/cache.go — cleanup() goroutine has no Stop mechanism → DP-169
**File:** pkg/optimization/cache.go:35-38 (stop chan), ~line 133 (cleanup goroutine)
**What:** Cache.NewCache() spawns a background `cleanup()` goroutine via `go c.cleanup()`. The `stop` channel is created but never sent to. No `Stop()` or `Close()` method exists. If the cache is garbage collected, the goroutine leaks.
**Why it matters:** Each cache instance leaks a goroutine. Currently only used in examples/benchmarks, not production — but if wired into prod, each game world would leak one goroutine per cache forever.
**Source:** analysis — goroutine leak risk
**Suggested fix:** Add `Stop()` method that closes the stop channel, and `defer cache.Stop()` at each call site.
