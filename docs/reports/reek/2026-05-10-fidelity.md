## CRITICAL

- **Severity / Status:** CRITICAL — DIVERGED
  **function:** `perform_violence()` (C: fight.c:1895) → `CombatEngine.PerformRound()` (engine.go:174) + `MakeHit()` (fight_core.go:335)
  **divergence:** Two separate hit-resolution code paths exist. `MakeHit()` faithfully reimplements C's `hit()` (THACO, str/dex apps, weapon type, backstab/circle/disembowel, minusdam, position multiplier, script trigger). `PerformRound().processCombatPair()` (engine.go:211) uses its own simplified `CalculateHitChance()` + `CalculateDamage()` that bypasses `MakeHit()` entirely. When mob AI initiates combat via `StartCombat()`, the combat engine processes rounds through the simplified path. When players use `doHit`/`doBackstab`, they go through the game layer's `doDamage()`/`MakeHit()` path. The same fight ends up resolved differently depending on who initiated it.
  **impact:** Mob-initiated fights use simplified damage (no INT/WIS THACO bonus, no position multiplier, no `get_minusdam` for weapon attacks, no skill-specific damage). Player-initiated fights use the full C logic. Players fighting mobs will see inconsistent damage numbers.
  **file:** pkg/combat/engine.go:211, pkg/combat/fight_core.go:335, pkg/combat/formulas.go:285-345
  **suggested fix:** `processCombatPair()` should call `MakeHit()` instead of its own `CalculateHitChance()` + `CalculateDamage()`. The engine loop should delegate to fight_core's complete hit logic.

- **Severity / Status:** CRITICAL — MISSING
  **function:** `load_messages()` (C: fight.c:125)
  **go port:** MISSING — no equivalent exists
  **divergence:** The C function reads `lib/misc/MESS_FILE` to populate `fight_messages[]` — a table of attack-type-specific death/hit/miss/god messages for skill_message(). The Go `SkillMessage` hook delegates to `SkillMessageFunc` but nothing loads the message file. The `DamMessage` function was reimplemented with a Go-specific threshold table that doesn't match the C ones (different damage tiers, missing higher tiers like EVISCERATES/DESTROYS/ROCKS).
  **impact:** Skill/spell messages use stubs instead of the full Dark Pawns message file. The weapon damage message table in Go has 14 tiers while C has 12 — the thresholds don't align.
  **file:** pkg/combat/fight_core.go:556-585 (DamMessage tiers), pkg/combat/fight_core.go:654 (SkillMessage)
  **suggested fix:** Either port the message file loading or tune the Go `DamMessage` tiers to match C exactly. Align `damage` threshold tiers with C fight.c: `{0,1,3,5,7,11,18,26,36,48,60,80,101,10000}` instead of C's `{0,1,3,5,7,11,15,19,23,33,43,53}`.

## HIGH

- **Severity / Status:** HIGH — DIVERGED
  **function:** `damage()` (C: fight.c:1210) → `TakeDamage()` (fight_core.go:192)
  **divergence:** The `group_gain()` gold-splitting logic (PRF_AUTOSPLIT, PRF_AUTOGOLD) is duplicated between `TakeDamage()` and `GroupGain()`. In C, gold handling is only in `group_gain()`. In Go, `TakeDamage()` has its own inline gold splitting at lines 380-430 that duplicates the C group_gain gold logic, AND `GroupGain()` is also called separately. This means gold may be distributed twice for NPC kills.
  **impact:** Gold duplication — group members may receive gold twice for the same kill.
  **file:** pkg/combat/fight_core.go:380-430 (inline gold in TakeDamage), fight_core.go:470-500 (GroupGain)
  **suggested fix:** Remove the inline gold splitting from TakeDamage and let GroupGain handle it exclusively, matching C's architecture.

- **Severity / Status:** HIGH — DIVERGED
  **function:** `perform_violence()` parry/dodge (C: fight.c:1927-1965) → `PerformRound()`/`processCombatPair()` (engine.go:174,211)
  **divergence:** In C, parry and dodge are checked inside `perform_violence()` at the attack-pair level — ONLY if the defender's fighter is also targeting them (FIGHTING(FIGHTING(ch)) == ch). A parried fighter gets `IS_PARRIED` set, which reduces their attacks. In Go, `MakeHit()` checks parry/dodge as part of the hit resolution, but `processCombatPair()` checks them separately via direct calls to `CheckParry()`/`CheckDodge()`. Since `MakeHit()` also has its own parry/dodge check, parry/dodge can fire TWICE for the same attack when going through `MakeHit()`.
  **impact:** Parry/dodge double-checked — double the expected success rate when MakeHit is called from the engine path.
  **file:** pkg/combat/fight_core.go:365-380 (MakeHit parry/dodge), engine.go:231-260 (processCombatPair parry/dodge)
  **suggested fix:** Remove the parry/dodge checks from MakeHit() and only keep them in processCombatPair(). Or wire processCombatPair() to call MakeHit() exclusively.

- **Severity / Status:** HIGH — DIVERGED
  **function:** `set_fighting()` / `stop_fighting()` (C: fight.c:255, 257) → `CombatEngine.StartCombat()`/`StopCombat()` (engine.go:139, 155)
  **divergence:** In C, `stop_fighting()` searches the combat list to reassign fighters when the target dies (finding another fighter who was fighting the same target). In Go, `StopCombat()` simply deletes the pair from the map — no reassignment. When a mob dies, other mobs fighting the same target don't get re-linked.
  **impact:** Multi-mob fights: if mob A dies, mobs B and C fighting the same player don't get their fighting target auto-reassigned. Player may stop being in combat despite still being in a fight. The C code specially handles `next_combat_list` to avoid losing the linked list during iteration. Go has no equivalent.
  **file:** pkg/combat/engine.go:155-169
  **suggested fix:** After removing a pair, check if the defender has other attackers and reassign their fighting target.

## MEDIUM

- **Severity / Status:** MEDIUM — DIVERGED
  **function:** `raw_kill()` (C: fight.c:595) → `RawKill()` (fight_core.go:620)
  **divergence:** Go `RawKill()` is missing two C behaviors:
  1. Tattoo clearing: C removes tattoos via `tattoo_af(ch, FALSE)`. Go skips this entirely.
  2. AFF_WEREWOLF/AFF_VAMPIRE removal: C removes these affect flags before make_dust/corpse. Go skips them.
  3. MOB_MEMORY/HUNTING clearing: C iterates `character_list` to `forget()` and `set_hunting(NULL)` for all mobs that remember the victim. Go skips this — kills leave stale hunt/memory entries in every NPC.
  **impact:** Tattoos persist after death. Werewolf/vampire affect flags persist. Every mob that remembered the victim still remembers a dead character.
  **file:** pkg/combat/fight_core.go:620-645
  **suggested fix:** Add `RemoveAffect` calls for AFF_WEREWOLF and AFF_VAMPIRE. Add tattoo clearing hook. Add global memory/hunting cleanup hook.

- **Severity / Status:** MEDIUM — DIVERGED
  **function:** `counter_procs()` (C: fight.c:1152) → `CounterProcs()` (fight_core.go:476)
  **divergence:** Go faithfully reproduces the C fallthrough bug where ALL three stats (HP/Mana/Move) get +1 on major milestones instead of random one. This is correct for fidelity. However, Go adds `+1` to ALL three stats regardless of the C switch fallthrough. In C: `case 1: GET_MAX_HIT++; case 2: GET_MAX_MANA++; case 3: GET_MAX_MOVE++; default: GET_MAX_HIT++; break;` — the fallthrough means: case 1000 → hit++, then mana++, then move++, then hit++ again. So HP gets +2, Mana gets +1, Move gets +1. Go gives all three +1 (missing the extra HP).
  **impact:** Players get 1 less max HP on major kill milestones. Small but intentional C behavior lost.
  **file:** pkg/combat/fight_core.go:518-520
  **suggested fix:** In the major milestones block, increment HP twice to match the C fallthrough: `IncreaseMaxStat(name, "hp")` called twice, or AddMaxHP(name, 2) style.

- **Severity / Status:** MEDIUM — DIVERGED

The game-layer version `GetMinusDam()` is never called from the combat engine — it's only used by `MakeHit()` path. A fix to one will silently drift from the other.
  **impact:** Silent divergence over time. Fixes to the AC damage formula only apply to one code path.
  **file:** pkg/combat/formulas.go:218-360, pkg/game/deferred_fight_fns.go:16-90
  **suggested fix:** Delete the duplicate in deferred_fight_fns.go and route all callers through the combat package's version.

### Key structural issue

The biggest finding is the **dual hit-resolution path**: `processCombatPair()` in engine.go resolves hits differently than `MakeHit()` in fight_core.go. The `MakeHit()` function is the faithful port of C's `hit()` — including all the THACO math, stat modifiers, parry/dodge integration, position multipliers, and skill-specific damage. But the engine's own tick loop bypasses it entirely, using simplified `CalculateHitChance()` + `CalculateDamage()` instead. This means mob-vs-player combat through the engine ticker uses different math than player-vs-mob combat through command handlers.

The intent was clean: put the full hit logic in `MakeHit()` and have the engine loop call it. But the engine loop was written independently and never wired to use it.
