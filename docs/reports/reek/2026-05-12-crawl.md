## Deep Dive: `pkg/game/` — Game Loop, State Management, Player Actions

### CRITICAL

**1. ActiveAffects data race — three inconsistent locking regimes**

`p.ActiveAffects` (player.go:72) is accessed under three different mutex conventions across the codebase, making concurrent reads/writes a guaranteed data race under any real load.

**file:line** — pkg/game/affect_update.go:47 (reads under w.mu.RLock), pkg/game/player_affects.go:8 (reads under p.mu.RLock), pkg/game/other_character.go:48 (no lock), pkg/game/follow.go:242 (removeCharmAffect, no lock), pkg/game/save.go:246 (reads under p.mu.RLock), pkg/session/affects_informative.go:17 (no lock), pkg/session/eat_cmds.go:66 (no lock), pkg/session/wiz_system.go:287 (no lock)

**What:** Player.ActiveAffects is accessed under w.mu in affect_update.go, under p.mu in player_affects.go/save.go, and under no lock at all in follow.go, other_character.go, and multiple session files. Every access path must agree on a single mutex — they don't.

**Why it matters:** Two concurrent goroutines can mutate the same slice header under different locks. The scheduler wakes the wrong one and you get torn writes, nil dereferences, or silently lost affects. Affects manage poison, haste, blindness, sanctuary — losing one of these mid-fight is a game bug, not a cosmetic one.

**Suggested fix:** Pick p.mu as the canonical lock for ActiveAffects (it's already the Player's own mutex, which is the correct owner). audit every read/write site and add the appropriate p.mu.RLock/p.mu.Lock. affect_update.go must use p.mu, not w.mu, for ActiveAffects access.

### HIGH

**1. removeCharmAffect mutates ch.ActiveAffects with zero locks**

**file:line** — pkg/game/follow.go:242-254

**What:** The `removeCharmAffect` helper function reads `ch.ActiveAffects` in a loop and performs slice mutation (`append(..[:i], ..[i+1:]...)`), but never acquires `ch.mu`. It's called from `StopFollower` (follow.go:115), which also holds no player lock at that call site.

**Why it matters:** Any concurrent access to the player's ActiveAffects (affect tick, save, spells ticking, examine command) while this runs produces a data race on the slice header. Slice mutations during concurrent iteration are undefined behavior in Go — you get panics or corrupted state, not clean errors.

**Suggested fix:** Acquire `ch.mu.Lock()` at the start of `removeCharmAffect` and release on return. The caller `StopFollower` already does other work that may need the lock — either move the lock there or re-acquire in the helper.

**2. doOrder calls executeCommand which is a no-op stub**

**file:line** — pkg/game/combat_control.go:79 → pkg/game/damage_stubs.go:115

**What:** `doOrder` in combat_control.go calls `w.executeCommand(vict, message)` expecting the ordered NPC/player to actually perform the command. But `executeCommand` (damage_stubs.go:115) is a stub: `func (w *World) executeCommand(ch *Player, command string) bool { _ = ch; _ = command; return true }`. It returns true and does nothing.

**Why it matters:** The 'order' command fully parses the target and message, sends the right notifications to the room, then silently discards the commanded action. The ordering player sees "Ok." but nothing happens. This is a functional gap — the order system looks complete but doesn't work.

**Suggested fix:** Wire `executeCommand` to actually dispatch the command string through the command registry (`cmdRegistry.Execute` or the session command handler). The target player/NPC needs to be found and their session/command handler invoked.

### MEDIUM

**1. affect_update.go guards Player fields with World mu instead of Player mu**

**file:line** — pkg/game/affect_update.go:44-70

**What:** `AffectUpdate()` reads `p.ActiveAffects` under `w.mu.RLock()` and replaces it under `w.mu.Lock()`. This means ActiveAffects is protected by two different mutexes depending on which code path accesses it.

**Why it matters:** Same root cause as the CRITICAL ActiveAffects issue. Even if all other sites migrated to p.mu, affect_update.go still uses w.mu — so the race persists from one side. The World mu should only guard world-level maps (players map, activeMobs map, rooms map), not individual player fields.

**Suggested fix:** In affect_update.go, hold w.mu.RLock only to snapshot the players map (which it already does correctly). Then iterate players and hold p.mu.Lock for each player's ActiveAffects access. The current code holds w.mu for far too long and locks the wrong granularity.

**2. executeMobCommand TOCTOU race between mob lookup and use**

**file:line** — pkg/game/world.go:458-470

**What:** `executeMobCommand` iterates `w.activeMobs` under `w.mu.RLock()` to find the mob by VNum, then releases the lock. After the lock is released, it checks `mob.IsAlive()` and uses the mob. Between the lock release and the use, the mob can be extracted from the world (killed, removed by zone reset, etc.), making the pointer stale.

**Why it matters:** This is a use-after-find race. If the mob dies mid-round, `executeMobCommand` operates on freed/cleaned state. The IsAlive check at line 468 is a best-effort guard but the mob could die between the check and the switch statement below.

**Suggested fix:** Either (a) keep the RLock held through the mob's switch cases (risk: long-running mob commands block world writes), or (b) copy the mob's identity data under the lock and operate on the copy, or (c) add a generation counter to MobInstance and verify it hasn't changed after the lock is released.

**3. Zone dispatcher context cancellation pattern — G118 suppressed but still a leak**

**file:line** — pkg/game/zone_dispatcher.go:73

**What:** `startZoneLocked` creates child contexts with `context.WithCancel(zd.ctx)` and stores the cancel function in the zoneWorker. The `// #nosec G118` comment suppresses a linter warning about not calling the stored cancel function. When the parent dispatcher context is cancelled (Stop), the child contexts are cascaded, but individual zone workers may be stopped without their cancel being called.

**Why it matters:** Stored cancel functions that are never called directly leak contexts through the lifetime of the dispatcher. The parent cancel does cascade in theory, but if a zone is dynamically removed while the dispatcher stays running, the child context is orphaned.

**Suggested fix:** Call the stored cancel function explicitly whenever a zoneWorker is removed. The G118 suppression hides a real resource management concern — the cancel should be tracked and called on cleanup, not just cascaded from the parent.

**What:** Equipment slot serialization uses `goSlotToCWearPos(slot)` which returns a C-style wear position for the save format. If a slot doesn't map (ok=false), the locate value stays 0, which C interprets as "inventory." An equipped item silently becomes an inventory item on reload.
