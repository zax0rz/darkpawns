## CRITICAL

### 1. Graceful shutdown is a lie
- **File:** `cmd/server/main.go:100`, `cmd/server/main_web.go:170`
- **What:** SIGINT handler just prints "Shutting down..." No connection draining, no context cancellation, no waitgroup.
- **Why it matters:** Players lose state on every restart. In-flight WebSocket connections are dropped without cleanup.
- **Source:** analysis (shutdown)
- **Suggested fix:** `shutdownCtx, cancel := context.WithTimeout(context.Background(), 30*time.Second); server.Shutdown(shutdownCtx)`

### 2. HandleNonCombatDeathScriptable is a stub
- **File:** `pkg/game/world.go:637`
- **What:** Logs "player would die from non-combat damage" and returns. Poison, drowning, falling damage — all silently ignored.
- **Why it matters:** Players can't die from environmental damage. No death mechanics at all for non-combat sources.
- **Source:** analysis (game logic)
- **Suggested fix:** Implement death handling or remove the function entirely

### 3. Spawner.StartZoneResets() is a no-op
- **File:** `pkg/game/spawner.go:57`
- **What:** Method locks s.mu and returns nil. Body is empty — no zones are reset.
- **Why it matters:** World.StartZoneResets() calls spawner.ExecuteZoneReset(zone) directly, but the spawner-level StartZoneResets() misleads callers into thinking it does something. It doesn't.
- **Source:** analysis (dead code)
- **Suggested fix:** Either implement the method or delete it.

## HIGH

### 4. Lock ordering deadlock risk in spawner
- **File:** `pkg/game/spawner.go:80`
- **What:** ExecuteZoneReset calls s.CanSpawn() (locks s.mu). World.StartZoneResets holds w.mu.Lock() via GetAllZones(). No documented lock ordering.
- **Why it matters:** If GetAllZones is ever called inside CanSpawn, world deadlock on next zone reset.
- **Source:** analysis (race)
- **Suggested fix:** Document lock ordering: w.mu → s.mu, never s.mu → w.mu. Verify in code.

### 5. http.DefaultServeMux with no isolation
- **File:** `cmd/server/main.go:112`
- **What:** All routes registered on http.DefaultServeMux. Any imported package that also registers on DefaultServeMux conflicts silently.
- **Why it matters:** Route hijacking or silent failure on package update.
- **Source:** analysis (architecture)
- **Suggested fix:** Create a fresh `http.ServeMux` and use that.

### 6. Duplicated server entry points
- **File:** `cmd/server/main.go`, `cmd/server/main_web.go`
- **What:** Two main() functions with duplicated boot logic. main.go supports TLS + slog. main_web.go doesn't support TLS + uses log. Routes diverge.
- **Why it matters:** Any fix to one won't hit the other. Bug fixes, route changes, dependency updates must be mirrored.
- **Source:** analysis (architecture)
- **Suggested fix:** Share common server setup in a helper function.

### 7. Reentrant RLock in Scriptable methods
- **File:** `pkg/game/world.go:802+`
- **What:** Every "Scriptable" method acquires w.mu.RLock() independently. Lua scripts can trigger cascading calls that deadlock on reentrant RLock.
- **Why it matters:** Go's sync.RWMutex is not reentrant. Lua calling GetItemsInRoomScriptable from within a GetMobsInRoomScriptable path = deadlock.
- **Source:** analysis (race)
- **Suggested fix:** Split into internal (no-lock) and external (with-lock) variants.

### 8. Doors don't reset on zone reset
- **File:** `pkg/game/spawner.go:150`
- **What:** Door state changes in zone resets are a TODO. Doors never reset their state.
- **Why it matters:** Players who lock a door keep it locked forever until server reboot.
- **Source:** analysis (game logic)
- **Suggested fix:** Implement the TODO for door state changes.

### 9. Server uses non-TLS HTTP by default with warning only
- **File:** `cmd/server/main.go:134-139`
- **What:** Falls back to HTTP with a warning log "not secure for production". No way to enforce TLS-only in config.
- **Why it matters:** Credentials and game data in cleartext over the wire.
- **Source:** analysis (security)
- **Suggested fix:** Fail closed if USE_TLS is not explicitly set.
