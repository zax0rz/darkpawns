## HIGH

### GroupGain: namedCombatant always reports IsNPC()=true — group XP never awarded

**File:** `pkg/combat/fight_core.go:593-600` (GroupGain → ApplyToGroupMembers → NewNamedCombatant → PerformGroupGain)

**What:** `GroupGain` creates a stub `namedCombatant` for each group member via `NewNamedCombatant(memberName, room)`. That stub's `IsNPC()` always returns `true` (line 1055: `func (n *namedCombatant) IsNPC() bool { return true }`).

**Why it matters:** `PerformGroupGain` guards `GainExp` behind `if !ch.IsNPC()` (line 577). Since the stub is always "an NPC", the check always fails. Group members never receive experience points from kills. The group leader gets their share but every member gets zero. Every group kill, every time.

**Source:** analysis

**Suggested fix:** Change `namedCombatant` to accept an `isNPC` field, or better, wire a lookup function that returns the true character data instead of a zero-value stub. Or move the XP gain logic to the game-layer hooks instead of relying on the stub.

```go
type namedCombatant struct {
    name  string
    room  int
    isNPC bool
}
func (n *namedCombatant) IsNPC() bool { return n.isNPC }
```

## MEDIUM

### Bash doesn't knock down — sets victim to PosFighting (highest stance)

**File:** `pkg/game/combat_melee.go:72` — `vict.SetPosition(combat.PosFighting)`

**What:** When bash succeeds, the victim is set to `PosFighting` (7) — the highest standing position. The comment says "Victim is bashed down" but the code does the opposite. Bash should set the victim to `PosSitting` (6) or below.

**Why it matters:** Bash is supposed to knock the target to the ground, making them vulnerable. Currently it does nothing to position. This is a gameplay bug: bash is a waste of skill points.

**Source:** analysis — comparison with C fight.c bash handling

**Suggested fix:** `vict.SetPosition(combat.PosSitting)` or `combat.PosResting` for the knockdown effect.

### Skill messages broadcast to room 0 instead of actual room

**File:** `pkg/combat/skill_messages.go:292` — `BroadcastMessage(0, roomMsg, ...)`

**What:** `InitSkillMessages` calls `BroadcastMessage(0, roomMsg, ...)` with hardcoded roomVNum = 0 because the function only receives name strings, not room numbers.

**Why it matters:** Skill messages (backstab, kick, bash, etc.) are broadcast to room 0, which is either wrong or a no-op if 0 is an invalid room. The actual combat room never sees skill-specific messages. Players only see generic "hits you for X damage" from the engine tick, not flavorful skill messages.

**Source:** analysis

**Suggested fix:** Accept a `roomVNum` parameter in the `SkillMessageFunc` callback signature, or extract room from the game-layer context when the callback is wired.

### MakeHit duplicates CalculateHitChance THAC0 logic — two divergent hit formulas

**File:** `pkg/combat/fight_core.go:759-830` (MakeHit) AND `pkg/combat/formulas.go:214-262` (CalculateHitChance)

**What:** `MakeHit()` recomputes THAC0 from scratch (getTHAC0, strApp, hitroll, INT, WIS, blessed weapon) — duplicating the full formula that `CalculateHitChance()` already implements in formulas.go. The engine tick uses `CalculateHitChance` (lighter path, no backstab/circle/position multiplier). The game-layer uses `MakeHit` (full path). They should converge on one formula.

**Why it matters:** Already flagged as CRIT-009 in the tracker (dual hit path). This is the structural cause. Two versions of the same formula means they'll drift over time as one gets fixed and the other doesn't.

**Source:** analysis — structural

**Suggested fix:** `MakeHit` should call `CalculateHitChance` for the to-hit roll, then apply skill-specific modifiers (backstab mult, disembowel, position damage bonus) on top.

### startCombatBetween doesn't register with CombatEngine

**File:** `pkg/game/combat_advanced.go:326-350`

**What:** `startCombatBetween` sets `SetFighting()` on both participants but never calls `CombatEngine.StartCombat()`. The game-layer fighting state and the engine-layer combat pair map are completely independent.

**Why it matters:** If combat is started through game-layer commands (assist, hit, kill) but not registered with the engine, the engine tick won't process those pairs. Combat relies on the engine tick for damage application. Multi-mob fights and follow-up rounds may silently not happen.

**Source:** analysis — integration seam
