## HIGH (5 confirmed, 1 needs context)

### CONFIRMED — HIGH-001: Lock ordering deadlock risk in spawner
- **File:** `pkg/game/spawner.go:80`
- **Severity:** HIGH
- **Daeron notes:** ExecuteZoneReset doesn't hold s.mu (correct), but calls CanSpawn which acquires s.mu.RLock(). World.StartZoneResets calls GetAllZones (w.mu.RLock()) then ExecuteZoneReset. Lock ordering: w.mu → s.mu. Fragile contract — no enforcement, just a comment.
- **Recommendation:** Document lock ordering in code comment. Consider adding a lock-ordering assertion helper.
- **Status:** OPEN

### CONFIRMED — HIGH-002: DefaultServeMux with no isolation
- **File:** `cmd/server/main.go:112`
- **Severity:** HIGH
- **Daeron notes:** main.go registers all routes on http.DefaultServeMux. main_web.go correctly creates http.NewServeMux(). Any imported package that registers on DefaultServeMux conflicts silently. Route hijacking or silent failure on package update.
- **Recommendation:** Switch main.go to use a fresh ServeMux like main_web.go does.
- **Status:** OPEN

### CONFIRMED — HIGH-003: Duplicated server entry points
- **File:** `cmd/server/main.go`, `cmd/server/main_web.go`
- **Severity:** HIGH
- **Daeron notes:** Two main() functions with duplicated boot logic. main.go uses slog + TLS support. main_web.go uses log + no TLS. Routes diverge (main.go has /metrics, main_web.go has /onboarding). Any fix to one won't hit the other.
- **Recommendation:** Extract shared boot logic into a helper. Keep entry points thin.
- **Status:** OPEN

### NEEDS CONTEXT — Reentrant RLock in Scriptable methods
- **File:** `pkg/game/world.go:802+`
- **Severity:** MEDIUM (requires concurrent Lock to deadlock)
- **Daeron notes:** Go's RWMutex allows multiple RLock holders. Reentrant RLock from the same goroutine only deadlocks if another goroutine calls Lock() concurrently. Low probability but real under load. Need to verify if Lua scripts can actually trigger cascading Scriptable calls from the same goroutine.
- **Recommendation:** Profile under load to see if this actually triggers. If Lua scripting is single-threaded per invocation, risk is minimal.
- **Status:** OPEN

### CONFIRMED — HIGH-004: Doors don't reset on zone reset
- **File:** `pkg/game/spawner.go:150`
- **Severity:** HIGH
- **Daeron notes:** Door state changes (D command) just log and do nothing. Players who lock a door keep it locked forever until reboot. This is a game logic gap — zone resets should restore door states.
- **Recommendation:** Implement the D command handler to restore door states per zone reset data.
- **Status:** OPEN

### CONFIRMED — HIGH-005: Non-TLS default with warning only
- **File:** `cmd/server/main.go:134-139`
- **Severity:** HIGH
- **Daeron notes:** Falls back to HTTP with `slog.Warn("Starting HTTP server (not secure for production)")`. No way to enforce TLS-only. Credentials and game data travel in cleartext.
- **Recommendation:** Fail closed if USE_TLS is not explicitly set, or at minimum make it configurable.
- **Status:** OPEN
