## HIGH

**SpellBless missing second affect application**
- **File:** `pkg/spells/affect_spells.go:33-37`
- **What:** SpellBless creates and applies `AFFECT_HITROLL +2` via `applyAffect()`, then creates `ApplySavingSpell -2` but never calls `applyAffect()` for the second affect — the second `aff` variable is overwritten silently.
- **Why it matters:** Bless grants both hitroll/+2 and saving spell/-2 in the C original. The Go version only grants half the spell effect. Players blessed in Go get no saving throw bonus.
- **Source:** analysis (code review)
- **Suggested fix:** Add `applyAffect(victim, aff)` after line 36.

**inflictDamage doesn't trigger death**
- **File:** `pkg/spells/damage_spells.go:164-182`
- **What:** `inflictDamage()` reduces HP to 0 but never checks if the victim died. No death handler is called. The `_ = dam` on line 180 confirms the unused parameter.
- **Why it matters:** Spells damage NPCs/players to 0 HP but they remain alive. No corpse, no death message, no room echo. Combat spells are purely cosmetic in their damage application.
- **Source:** analysis (dead code, error handling)
- **Suggested fix:** After `d.SetHP(newHP)`, check if `newHP == 0` and call world death handlers (HandleNonCombatDeath or handleMobDeath).

## MEDIUM

**AddItemToRoom doesn't track Location — still called by spells**
- **File:** `pkg/game/world_object.go:27` (deprecated), `pkg/game/world_scriptable.go:128` (caller), `pkg/spells/affect_spells.go:castCoC` (caller)
- **What:** `AddItemToRoom()` only appends to the roomItems slice without setting `obj.Location` or `obj.RoomVNum`. When spells like castCoC and castSilkenMissile call this, the spawned objects have no valid Location, so ExtractObject can't find them to clean up. The deprecated comment warns about this.
- **Why it matters:** Objects leaked into rooms via spell-cast code paths will have broken location tracking. Gate extraction, zone resets, or timer expiration on these objects will silently fail.
- **Source:** analysis (code review)
- **Suggested fix:** Replace all `AddItemToRoom()` calls in spell code with `MoveObjectToRoom()` which properly sets Location and RoomVNum.

**Spell routine stubs — MagGroups, MagMasses, MagAreas, MagSummons, MagCreations, MagAlterObjs**
- **File:** `pkg/spells/affect_spells.go:205-250`
- **What:** All six spell routine dispatch functions are no-op stubs accepting parameters via `_ =` and doing nothing. SetupSpellInfo registers spells with these routines (e.g., SpellBurningHands has RoutineGroups?), but when CallMagic dispatches to them, nothing happens.
- **Why it matters:** Group heal (SpellMassHeal), area effects from CallMagic path, group invis, mass recall — any spell relying on these routine dispatchers is completely non-functional.
- **Source:** analysis (dead code)
- **Suggested fix:** Implement each routine or set up the SpellInfo for these spells to use RoutineManual instead, then implement the manual spell.

**SpellGate rawKill attack type mismatch**
- **File:** `pkg/game/gates.go:155-167`
- **What:** `RawKill()` maps attack type strings to integer constants (slash→1, stab→2, etc.), but the `rawKill()` call at line 168 uses the integer `at` as the attack type. The original C `raw_kill()` passes `TYPE_SUFFERING` (a special constant), not a weapon damage type. The switch only handles physical attack types — `"blast"` is not mapped, so `at` stays 0.
- **Why it matters:** Gate collision death uses attack type 0 instead of TYPE_SUFFERING. Death messages and logging will show incorrect attack type.
- **Source:** analysis (fidelity)
- **Suggested fix:** Add `case "blast": at = TYPE_SUFFERING` (or match the C constant).

**SpellSilkenMissile constant may conflict**
- **File:** `pkg/spells/spells.go:103`
- **What:** `SpellSilkenMissile = 200` is noted as "DP original — not in C spell_parser spello table". The spell system uses VNUM-based spell IDs from 1-207 in C. Assigning 200 for a custom spell may overlap with future C-ported spells.
- **Why it matters:** If/when spells with IDs 200-206 are ported from C (AcidBreath at 205, etc.), silken missile will conflict. Currently there's no runtime collision since only 200-206 are assigned, but 201-207 are also used for breath weapons.
- **Source:** analysis (code review)
- **Suggested fix:** Move DP custom spells to a high range (500+) to avoid collision with C port space.
