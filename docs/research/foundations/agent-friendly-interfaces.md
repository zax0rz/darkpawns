# Architecting the Human-AI Sandbox: Agent-Friendly Interfaces for Multi-User Dungeons

The integration of Large Language Models (LLMs) and autonomous agents into multiplayer text-based environments represents a paradigm shift in game design and artificial intelligence research. Historically, Multi-User Dungeons (MUDs) and Interactive Fiction (IF) systems were engineered exclusively for human consumption, relying on rich natural language prose to convey state and requiring free-form text input for actions. As game engines evolve to support environments where artificial intelligence agents participate as first-class citizens alongside human players, a fundamental architectural tension emerges. Human players require immersive, narrative-driven prose, whereas autonomous agents operate most efficiently and deterministically on strictly typed, structured data. 

The development of modern mixed-participation environments requires a radical departure from monolithic text parsers. Building a system such as *Dark Pawns*—where human players and autonomous agents must share a single, coherent game state without friction—demands an architecture that decouples the underlying simulation from the presentation layer. This comprehensive technical research document investigates the design of agent-friendly interfaces for modern text-based game systems. Drawing upon legacy MUD protocols, contemporary academic frameworks, and state-of-the-art agent communication standards, the analysis is divided into three critical domains: the representation of the game state, the design and discoverability of the action space, and the underlying dual-output server architecture required to synchronize the human and machine experience.

## Part 1: Structured Game State Output

The foundation of any autonomous agent framework is the observation phase of its operational loop. How a game server represents the state of the world directly dictates an agent's ability to reason, plan, and execute actions. In a mixed-participation MUD, presenting this data requires navigating the profound computational trade-offs between traditional prose and highly structured data schemas.

### The Parsing Problem: Prose Versus Structured State

Traditional MUDs output raw strings of descriptive text. A standard spatial observation might read: "A foot or so of ordinary tile floor is at your feet; but beyond that, the corridor is cut away... You see a goblin here. Exits are North and West" [cite: 1]. While modern LLMs are remarkably adept at Natural Language Understanding (NLU), forcing an agent to parse game state entirely from literary prose introduces several critical failure points for autonomous operation. 

First, parsing requires the agent to expend valuable inference cycles and context window capacity to extract entities, relationships, and spatial topology from descriptive text. Second, reliance on narrative introduces hallucination and ambiguity. Literary descriptions often employ metaphors or atmospheric text that an LLM might interpret as actionable entities; for instance, a description stating that "shadows dance on the walls" might prompt a less sophisticated agent to attempt to attack the "shadows." Third, prose often relies on human object permanence and common sense. If a door is described as "heavy and locked" in one turn, but omitted in the next turn's brief combat update, an agent may incorrectly assume the door no longer exists.

Research involving agents operating in complex simulations demonstrates that environment legibility is paramount [cite: 2]. In experiments where LLM agents were tasked with playing management simulations, researchers found that models struggled immensely with text-based spatial renderings, such as ASCII maps, but thrived when provided with clean, well-structured omniscience via interfaces that replicated data points directly as code or structured schemas [cite: 2]. Topologies and graphs are vastly superior to geometric or narrative representations for machine consumption, as they eliminate the need for the agent to perform intermediate Named Entity Recognition (NER) before deciding on a strategy.

### Out-of-Band Communication and Hybrid Approaches

The MUD development community previously encountered and solved the problem of separating data from prose through Out-of-Band (OOB) protocols like the Generic Mud Communication Protocol (GMCP) and the Mud Server Data Protocol (MSDP) [cite: 3, 4]. Originally designed to allow graphical MUD clients to update health bars, inventory screens, or minimaps without scraping the main text feed, these protocols operate by sending JSON-formatted data hidden within Telnet sub-negotiation channels [cite: 4, 5].

A standard GMCP exchange utilizes the byte sequence format `IAC SB GMCP <package> <data> IAC SE` [cite: 4]. For instance, a game server might broadcast room information as a structured JSON object. The Aardwolf MUD schema outputs room topologies as: `Room.Info { "num": 16542, "name": "A Quiet Section of the Swamp", "zone": "fens", "exits": { "n": 16543, "s": 16545 } }` [cite: 6]. Similarly, character vitals are pushed continuously without cluttering the combat log: `Char.Vitals { "hp": 120, "maxhp": 150, "mana": 45 }` [cite: 7]. Other implementations, such as those found in Iron Realms Entertainment (IRE) games, expand this to include detailed arrays of afflictions, active defenses, and target status [cite: 7].





For hybrid environments, implementing a system inspired by GMCP or modern WebSockets allows the server to serve two distinct masters. Human clients receive standard Telnet or WebSocket streams where structured data drives their graphical UI overlays while they read the prose. Conversely, an AI agent's client can be configured to completely suppress the prose stream, subscribing exclusively to the JSON packages to maintain its internal state model. This hybrid approach mirrors the findings of the CALYPSO research project, which utilizes LLMs as Dungeon Masters' Assistants. In the CALYPSO paradigm, the LLM utilizes a discrete set of tools to fetch background knowledge and structured game rules independently from the prose it subsequently generates for the players, effectively separating the factual state from the narrative output [cite: 8].

### Essential Fields and Object Tree Schemas

When designing the JSON schema for an AI-first MUD interface, the data must be highly relational. Academic testbeds provide excellent blueprints for this data architecture. Environments like *DiscoveryWorld* abstract game environments into "object trees" or knowledge graphs [cite: 9]. In these environments, every room or space is represented as a tile grid (e.g., a 32x32 configuration), and all objects at a given tile are represented by an object tree where the root node contains physical structures (such as a table), and child nodes contain their contents (such as a jar on the table) [cite: 9]. 

This methodology is echoed in the PETSCIITile framework and other TILE-based academic research projects, which demonstrate that agents perform far better when an environment is converted from ASCII or narrative descriptions into structured JSON tilemaps [cite: 10, 11]. By enforcing a data-driven interface between the design of the environment and the code that executes it, context overflow during inference is drastically mitigated [cite: 10]. To provide an LLM with sufficient context without overwhelming its token limits, the observation schema must include specific essential fields.

| Category | Schema Components | Agent Utility | Example JSON Signature |
| :--- | :--- | :--- | :--- |
| **Topology & Navigation** | Room ID, localized coordinates, terrain properties, and enumerated dictionaries of egress points mapped to adjacent node IDs. | Enables deterministic pathfinding and graph traversal algorithms, bypassing spatial hallucination. | `{"id": "rm_102", "terrain": "swamp", "exits": {"north": "rm_103"}}` |
| **Entity Recognition** | Strictly typed arrays of visible actors. Includes unique IDs, base types (NPC, player), alignment, and exact health percentages. | Provides exact, disambiguated target IDs for combat and interaction logic. | `{"id": "mob_442", "name": "goblin", "stance": "hostile", "health": 80}` |
| **Inventory & Equipment** | Granular data on encumbrance limits, nested container logic (items within backpacks), and current equipment loadouts. | Allows the agent to calculate optimal item usage, gear swapping, and looting constraints. | `{"capacity": 50, "equipped": {"main_hand": "itm_sword"}, "held": []}` |
| **Status & Constraints** | Active buffs, debuffs, resource pools (Mana, Stamina), and exact millisecond values for active timers/cooldowns. | Critical for survival logic and action pacing, preventing the agent from initiating invalid tool calls. | `{"mana": 45, "afflictions": [{"name": "poison", "duration_ticks": 3}]}` |

By structuring the environment as a graph of JSON objects, the LLM can effortlessly map the state into its working memory. Frameworks like *TextArena*, which benchmarks agentic behavior across dozens of text-based games, rely entirely on sending these structured state updates step-by-step, providing observations and rewards that enable models to refine their strategies dynamically [cite: 12, 13].

## Part 2: Action Space Design

If the game state is the agent's sensorium, the action space represents its actuators. The design of how an agent interacts with a game dictates the complexity of behaviors it can exhibit and the computational overhead required to execute them safely. For artificial intelligence agents, the transition from unstructured natural language to structured action invocation is the most critical hurdle in game integration.

### The Fallacy of Natural Language Commands

In classical Interactive Fiction and MUDs, the action space is functionally infinite but mechanically narrow. Players type free-form text, and the server utilizes a complex Natural Language Processing (NLP) parser to extract verbs, prepositions, and target objects. The server attempts to map inputs like "put the red key in the brass lock" to underlying code execution [cite: 14]. 

For LLM agents, generating free-form text to control a game is an anti-pattern that leads to severe performance degradation. Research using the *Jericho* framework—a suite of classic interactive fiction games designed to measure AI performance—highlights that agents struggle immensely with the "guess the verb" problem [cite: 15, 16, 17]. Because the action space is technically the space of all possible words and combinations, agents routinely generate humorously nonsensical solutions or hallucinate commands that the legacy parser inevitably rejects [cite: 16, 17]. To mitigate this in academic settings, researchers using *TextWorld* and *Jericho* often rely on predefined "action templates" to artificially bound the natural language space into a smaller subset learnable by the agent's neural network [cite: 18, 19].

However, modern agent architectures abandon the text parser entirely in favor of strict tool-calling or function-calling paradigms. This mirrors the enterprise architectures of OpenAI's Function Calling or Anthropic's Model Context Protocol (MCP) [cite: 20, 21]. In a modern system, the server exposes actions not as a command-line prompt, but as a rigid API of discrete tools. Instead of the agent generating the string `"attack goblin"`, it executes a structured payload containing the method and the parameters. This allows the server to bypass NLP parsing entirely, directly mapping the agent's intent to the game engine's internal function signatures.

### Discoverability and Context Window Management

A major challenge in complex game environments is exposing the action space without overwhelming the LLM's context window. If a MUD features hundreds of possible commands ranging from combat abilities to crafting sequences and social interactions, sending the JSON schema for every possible action on every turn is computationally prohibitive, highly expensive, and dilutes the agent's attention mechanism.

The solution lies in "Dynamic Tool Excision," a pattern heavily utilized in Model Context Protocol (MCP) implementations. MCP is an open standard designed to connect LLM applications seamlessly with external data sources and tools, providing context-sensitive actions that appear and disappear based on the localized situation [cite: 21]. 





In an MCP-architected MUD, such as the open-source *MUD-MCP* project, the action space is treated as dynamic, deeply contextual state [cite: 22]. The server actively monitors the agent's specific topological environment and issues continuous `notifications/prompts/list_changed` events [cite: 22]. For example, the `open_chest` tool is only injected into the LLM's system prompt when the agent enters a room containing an interactive chest [cite: 22]. Similarly, the entire suite of `battle` tools becomes available exclusively when hostile entities are present [cite: 22]. This surgical exposure ensures the LLM's available tool list perfectly mirrors its localized affordances, eliminating the risk of the agent hallucinating actions that are physically impossible in its current state.

| Action Space Paradigm | Input Mechanism | Constraint Communication | Context Window Impact |
| :--- | :--- | :--- | :--- |
| **Free-Form Text** | Natural Language Parser | Trial and error; server error strings. | Low, but error rate is exceedingly high. |
| **Action Templates** | Bounded Text Strings | Predefined grammar matrices. | Medium; templates scale poorly with game complexity. |
| **Static Tool Calling** | JSON-RPC | Hardcoded tool schemas in the system prompt. | Extremely High; overwhelming tool lists degrade reasoning. |
| **Dynamic MCP Tools** | Contextual JSON-RPC | Ephemeral schema injection via `list_changed` events. | Optimized; agent only processes immediately valid affordances. |

### Handling Constraints, Cooldowns, and Ambiguity

In multiplayer environments, actions are heavily constrained by game mechanics. Relying on an agent to learn constraints through reinforcement learning and trial and error is deeply inefficient for general-purpose LLMs interacting with a live server [cite: 23]. Instead, constraints must be communicated explicitly and preemptively within the action space schema.

Prerequisites and costs must be documented directly in the tool description provided to the LLM. If a spell requires resource expenditure, the schema must state the exact numerical requirement. This allows the LLM's internal reasoning engine to calculate whether it can afford the action based on its current state payload [cite: 7]. Timers and cooldowns should be handled simultaneously by active server routing and passive client filtering. The server broadcasts a status package that enumerates exactly how many milliseconds remain on specific action cooldowns [cite: 24]. Concurrently, the MCP server can temporarily revoke the specific tool from the agent's available context until the cooldown expires, acting as a firewall against invalid API calls.

Furthermore, moving to a structured JSON-RPC format solves the problem of action ambiguity. When human players type partial commands, traditional MUDs use substring matching algorithms to deduce intent [cite: 25]. For AI agents utilizing strictly typed unique identifiers, ambiguity is systematically eliminated. However, should an agent hallucinate an incorrect target identifier, the server must be engineered to return a structured JSON error containing rich semantic feedback, rather than a generic failure string. Returning an error object detailing why the action failed, alongside an array of valid targets currently in the room, allows the agent to ingest the correction and seamlessly self-correct on the subsequent inference pass.

## Part 3: Dual-Output Server Architecture

To effectively support both human players requiring rich prose and AI agents requiring structured data schemas, the game server must be constructed on a dual-output architecture. This ensures a Single Source of Truth (SSoT) for the underlying game state while completely decoupling the physics and rules engine from the presentation and transport layers.

### The Narrative Layer vs. The Data Layer

The core architectural requirement for an ecosystem like *Dark Pawns* is the strict separation between the data layer and the narrative layer. In legacy MUD architectures, the game engine calculates mechanical outcomes and immediately executes string concatenation functions to build the output. This tight, monolithic coupling of logic and presentation is catastrophic for mixed-client environments, as it requires agents to reverse-engineer the string back into data.

Instead, the game engine must operate purely as a state machine, generating strictly typed event objects. For instance, a combat interaction generates an event payload defining the source, target, action type, numerical value, and resultant state change. Once this event is committed to the authoritative state, it passes through an outbound routing layer that filters and formats the data based on the connected session's specific client type.

For human sessions, the event payload is passed to a templating engine that translates the data object into immersive, atmospheric prose, injecting variety and narrative flair before transmitting it over Telnet or a web client. For AI sessions, the prose generation layer is bypassed entirely. The event is serialized directly into JSON and pushed to the agent's interface. 

### Existing Architectures and Portal-Server Paradigms

Developing this complex routing architecture from scratch presents significant technical risk, but existing open-source MUD engines provide highly effective blueprints. 

The *Evennia* framework utilizes a robust two-process architecture that strictly separates the `Portal` from the `Server` [cite: 25]. In this paradigm, the Server process contains the database abstraction, the object trees, the command sets, and the core game loop [cite: 25]. It remains entirely ignorant of how players are connecting. The Portal process handles all external network connections, managing Telnet, WebSockets, and SSH interfaces, and understands protocol negotiation [cite: 25]. 

Within Evennia, a participant is represented by a `Session` object [cite: 26]. The Server sends protocol-agnostic data to the Session, and the Portal handles the final translation [cite: 25]. For a mixed-participation environment, developers can implement custom Session classes within the Portal. A human Session translates the incoming game events into text, while an AI Session negotiates a WebSocket connection and expects JSON-RPC packages, allowing the underlying Server to treat humans and AI identically in terms of core game logic.

| Architecture Component | Traditional MUD Engine | Evennia Portal-Server Paradigm | MCP-Integrated MUD (*EllyMUD*) |
| :--- | :--- | :--- | :--- |
| **State Management** | Tightly coupled with text generation. | Abstracted in the Server process via object trees. | Abstracted; exposed via REST and WebSocket APIs. |
| **Connection Handling** | Monolithic Telnet listener. | Handled by the Portal process (Telnet, Web, SSH). | Multi-port protocol listeners. |
| **Command Processing** | NLP text parsing against global command lists. | Extensible `CmdSets` attached to specific `Sessions`. | Virtual Sessions parsing structured JSON-RPC tool calls. |
| **Agent Support** | Requires screen-scraping and regex parsing. | Feasible via custom `Session` data formatting. | Native; features dedicated ports for AI testing and gameplay. |

Newer projects designed specifically for the modern web demonstrate the viability of native multi-protocol support. The *EllyMUD* project, built in Node.js and TypeScript, inherently supports multiple distinct connection methods [cite: 27]. It utilizes Telnet on port 8023 for legacy clients, WebSockets on port 8080 for web-based graphical interfaces, and notably features a dedicated Model Context Protocol (MCP) server running on port 3100 [cite: 27, 28]. This MCP interface is secured with API key authentication and provides "Virtual Sessions" specifically designed for AI agents to play autonomously and test deterministic game ticks, bypassing the standard character login flow designed for humans [cite: 27, 29].

### Transport Considerations: WebSockets and Real-Time Streaming

The choice of transport protocol is critical for ensuring synchronization between human and machine players in real-time environments. While legacy OOB protocols like GMCP rely heavily on Telnet sub-negotiation [cite: 4], modern AI agent frameworks generally expect robust HTTP or WebSocket transport mechanisms. 

For AI agents operating in dynamic environments, WebSockets provide the necessary full-duplex, low-latency communication required for real-time interaction. Unlike standard HTTP polling, which introduces unacceptable lag and server load, a WebSocket allows the server to instantly push state changes to the agent [cite: 27]. If an enemy enters a room, the server immediately pushes a topological update over the WebSocket. This triggers an asynchronous interruption in the agent's routine, allowing it to react to environmental shifts instantly. Because LLM inference inherently involves processing latency—often measured in Time To First Token (TTFT)—minimizing the network transport latency via persistent WebSocket connections ensures the agent is afforded the maximum possible time window to process the state change, reason about its implications, and respond with a tool call before the game engine advances to the next chronological tick.

## Conclusion and Strategic Recommendations

Designing an environment where AI agents are first-class participants alongside human players requires discarding legacy text-parsing architectures in favor of modern, data-driven API design. The game server must function as a true dual-output engine—operating as a literary simulator for human participants while simultaneously providing a strictly typed, structured data sandbox for machine intelligence.

Based on the synthesis of legacy MUD protocols, contemporary academic frameworks like TextArena, and modern agent standards, the following architectural recommendations are essential for achieving robust integration:

1. **Implement a Portal-Server Architecture:** Decouple the core game loop from the presentation layer. The engine must process strictly typed objects and events, routing these through distinct presentation layers based on the session type—a prose-generation engine for human interfaces, and a JSON serialization engine for agent connections.
2. **Adopt the Model Context Protocol (MCP):** Expose the game to AI clients via MCP over WebSockets. This standardizes the interface, allowing seamless integration with advanced reasoning models without requiring bespoke middleware.
3. **Utilize Dynamic Tool Excision:** Protect the LLM's context window by dynamically injecting and retracting tools from the agent's schema based on localized topological state. Agents must only be exposed to actions that are physically and logically possible in their immediate vicinity.
4. **Enforce Constraints Programmatically:** Map action cooldowns, resource costs, and mechanical prerequisites directly into the tool descriptions. Rely on server-side validation to reject invalid actions, and return rich, semantic JSON-RPC errors that enable the agent to self-correct its reasoning.
5. **Represent Spatial Geometry as Topology:** Avoid passing geometric maps or narrative descriptions of space to the agent. Structure the environment as a relational graph of object trees, allowing the LLM to leverage its inherent strengths in logical deduction rather than spatial processing.
